# Mihits 项目架构分析报告

> 分析日期：2026年6月17日  
> 分析版本：基于当前代码库

---

## 一、项目概述

**Mihits** 是一个基于 Qt/C++ 开发的在线五子棋对战平台，支持用户注册登录、联机对弈、人机对战和对局回放功能。

### 核心特性
- 用户系统（注册/登录，MD5加密）
- 联机对战（基于TCP/UDP网络通信）
- 人机对战（AI托管代打）
- 对局记录与回放
- 游戏大厅与房间系统

## 二、整体架构分析

### 2.1 架构模式

项目采用混合设计模式：

| 模式 | 应用位置 | 作用 |
|------|----------|------|
| **单例模式** | `CKernel::Get_Instance()` | 核心调度器全局唯一 |
| **中介者模式** | `INetMediator` | 网络通信的中介封装 |
| **观察者模式** | Qt信号-槽 | UI与业务逻辑解耦 |

### 2.2 模块划分

```
Mihits/
├── 核心层 (Core)
│   └── ckernel.*           # 核心调度器（单例）
├── UI层 (Presentation)

│   ├── logindialog.*      # 登录/注册界面
│   ├── fiveinlinezone.*   # 五子棋分区（房间列表）
```

## 三、功能模块详细介绍

### 3.1 用户系统

**功能：**
- 用户注册：手机号、昵称、密码
- 用户登录：手机号 + 密码
- 密码使用MD5加密传输

**实现文件：**
- `ckernel.cpp`: `slot_registerRq()`, `slot_loginRq()`
- `packdef.h`: `STRU_REGISTER_RQ`, `STRU_LOGIN_RQ`

**流程：**
```
用户注册 → MD5加密密码 → 打包发送 → 服务器验证 → 返回结果 → 提示用户
用户登录 → MD5加密密码 → 打包发送 → 服务器验证 → 返回用户ID和昵称 → 进入大厅
```

### 3.2 游戏大厅

**功能：**
- 显示游戏分区（目前只有五子棋）
- 点击进入分区

**实现文件：**
- `dialog.*`: 游戏大厅主界面
- `ckernel.cpp`: `slot_joinzone()`

### 3.3 五子棋分区

**功能：**
- 显示房间列表（120个房间）
- 定时刷新房间信息（每秒通过`QTimer`）
- 创建/加入房间
- 查看对局记录
- 回放单局记录

**实现文件：**
- `fiveinlinezone.*`: 分区界面
- `ckernel.cpp`: `slot_joinzone()`, `slot_roominfozone()`, `slot_dealzoneroominfo()`

**关键逻辑：**
```cpp
// 定时刷新房间信息
connect(&m_rqtimer, SIGNAL(timeout()), this, SLOT(slot_roominfozone()));
m_rqtimer.start(1000);  // 每秒刷新
```

### 3.4 房间系统

**功能：**
- 房间内有两个位置：host（黑方）和player（白方）
- 准备/取消准备
- 房主可以开始游戏（需要双方准备）
- 下棋（落子）
- AI托管（可以托管黑方或白方）
- 游戏结束判定

**实现文件：**
- `roomdialog.*`: 房间界面
- `fiveinline.*`: 五子棋游戏逻辑
- `ckernel.cpp`: 网络协议处理

**房间状态枚举：**
```cpp
enum ENUM_ROOM_STATUS {_host, _player, _spec};
// _host: 房主（黑方）
// _player: 玩家（白方）
// _spec: 观众
```

### 3.5 对局记录

**功能：**
- 查看所有对局记录
- 回放单局记录（作为观众）

**实现文件：**
- `record.*`, `recordlist.*`: 记录显示
- `ckernel.cpp`: `slot_sendrecordrq()`, `slot_deal_allrecordrs()`, `slot_deal_singlerecordrs()`

## 五、潜在缺陷分析

### ⚠️ A. 内存管理问题（严重性：高）

#### A.1 CKernel内存泄漏

**问题描述：**
- `CKernel`的析构函数为空：
```cpp
~CKernel(){}
```

- 虽然`Des_Instance()`中删除了部分指针，但并非所有动态分配的对象都被删除：

```cpp
void CKernel::Des_Instance()
{
    qDebug()<<__func__;
    delete m_dialog;        // ✅ 已删除
    delete m_client;       // ✅ 已删除
    delete m_logindialog;  // ✅ 已删除
    // ❌ m_fiveinlinezone 未删除
    // ❌ m_roomitem 未删除
}
```

**建议修复：**
```cpp
CKernel::~CKernel()
{
    delete m_dialog;
    delete m_logindialog;
    delete m_roomdialog;
    delete m_fiveinlinezone;
    delete m_roomitem;
    delete m_client;
}
```

#### A.2 roomdialog.cpp中的迭代器bug

**问题描述：**
- 第335-347行：`rmmem()`函数中，变量名从`it`变成了`ite`，这会导致编译错误

```cpp
void roomDialog::rmmem(int id)
{
    for(auto ite = userlist.begin(); ite!=userlist.end(); ite++)  // ❌ 应该是it
    {
        int curid = (*ite).first;
        if(curid==id)
        {
            ite = userlist.erase(ite);  // ❌ ite被erase后失效
            break;
        }
    }
}
```

**建议修复：**
```cpp
void roomDialog::rmmem(int id)
{
    for(auto it = userlist.begin(); it!=userlist.end();)
    {
        int curid = (*it).first;
        if(curid==id)
        {
            it = userlist.erase(it);  // ✅ 正确：使用erase的返回值
            break;
        }
        else
        {
            ++it;
        }
    }
}
```

#### A.3 智能指针使用建议

**建议：** 使用`std::unique_ptr`管理动态内存，避免手动delete：

```cpp
class CKernel : public QObject
{
    // ...
    std::unique_ptr<Dialog> m_dialog;
    std::unique_ptr<LoginDialog> m_logindialog;
    std::unique_ptr<roomDialog> m_roomdialog;
    // ...
};
```

### ⚠️ B. 网络协议安全性问题（严重性：高）

#### B.1 缓冲区溢出风险

**问题描述：**
- `packdef.h`中多个结构体使用固定大小的字符数组：
```cpp
#define _MAX_SIZE (40)

typedef struct STRU_REGISTER_RQ
{
    PackType type;
    char tel[_MAX_SIZE];      // 40字节
    char name[_MAX_SIZE];      // 40字节
    char password[_MAX_SIZE];  // 40字节
}STRU_REGISTER_RQ;
```

- `ckernel.cpp`中使用`strcpy`而不检查边界：
```cpp
void CKernel::slot_registerRq(QString tel, QString password, QString nick)
{
    STRU_REGISTER_RQ rq;
    strcpy(rq.tel, tel.toStdString().c_str());  // ❌ 可能溢出
    strcpy(rq.password, getmd5(password).c_str());
    std::string strname = nick.toStdString();
    strcpy(rq.name, strname.c_str());  // ❌ 可能溢出
    sendData((char*)&rq, sizeof(rq));
}
```

**建议修复：**
使用`strncpy`或`qstrncpy`：
```cpp
void CKernel::slot_registerRq(QString tel, QString password, QString nick)
{
    STRU_REGISTER_RQ rq;
    qstrncpy(rq.tel, tel.toStdString().c_str(), _MAX_SIZE);
    qstrncpy(rq.password, getmd5(password).c_str(), _MAX_SIZE);
    std::string strname = nick.toStdString();
    qstrncpy(rq.name, strname.c_str(), _MAX_SIZE);
    sendData((char*)&rq, sizeof(rq));
}
```

#### B.2 序列化/反序列化风险

**问题描述：**
- 直接使用C风格的结构体序列化（类型转换）
- 存在字节对齐问题
- 跨平台兼容性差（大小端问题）

```cpp
void CKernel::slot_deal_readydata(unsigned int lSendIP, char* buf, int nlen)
{
    int type = *(int*)buf;  // ❌ 没有检查buf长度
    // ...
    PFUN pf = NetPackMap(type);
    (this->*pf)(lSendIP, buf, nlen);  // ❌ 直接调用，没有验证数据完整性
}
```

**建议：**
- 使用序列化库（如Protocol Buffers, MessagePack, JSON）
- 或者添加数据长度和校验和字段

#### B.3 缺乏边界检查

**问题描述：**
- `ckernel.cpp`第133行：没有检查buf的长度
- 多个地方直接类型转换，没有验证数据长度

**建议修复：**
```cpp
void CKernel::slot_deal_readydata(unsigned int lSendIP, char* buf, int nlen)
{
    if(nlen < (int)sizeof(int))  // ✅ 添加长度检查
    {
        qDebug() << "Packet too short";
        delete [] buf;
        return;
    }
    int type = *(int*)buf;
    if(type >= _DEF_PACK_BASE && type < _DEF_PACK_BASE + _DEF_PACK_COUNT)
    {
        PFUN pf = NetPackMap(type);
        if(pf == nullptr)
        {
            qDebug() << "fun ptr not inititalized:" << type;
            delete [] buf;
            return;
        }
        (this->*pf)(lSendIP, buf, nlen);
    }
    else
    {
        qDebug() << "protocol types err";
    }
    delete [] buf;
}
```

**问题描述：**

**建议修复：**
使用枚举或常量：

```cpp
const int ZONE_FIVE_IN_LINE = 0x10;
```

**问题描述：**
- `_MAX_SIZE=40`：手机号和昵称可能需要更长
- `DEF_ZONE_ROOM_COUNT=121`：硬编码

**建议：**
- 增加`_MAX_SIZE`到合理值（如64或128）
- 使用配置文件或常量定义

**问题描述：**
- 网络操作缺乏错误处理
- 很多函数没有返回值检查

```cpp
void CKernel::sendData(char* buf, int nlen)
{

```

**建议：**
```cpp
bool CKernel::sendData(char* buf, int nlen)

    bool success = m_client->SendData(0, buf, nlen);
```

### ⚠️ D. 安全性问题（严重性：高）

#### D.1 MD5加密不安全

**问题描述：**
- MD5已经被证明是不安全的哈希算法
- 容易被彩虹表攻击

**建议：**
使用更安全的哈希算法，如SHA-256、bcrypt、Argon2等：

```cpp
#include <QCryptographicHash>

static std::string getSHA256(QString str)
{
    QByteArray ba = str.toUtf8();
    QByteArray hash = QCryptographicHash::hash(ba, QCryptographicHash::Sha256);
    return hash.toHex().toStdString();
}
```

#### D.2 密码传输缺乏盐值（salt）

**问题描述：**
- 虽然密码用MD5加密，但缺乏盐值
- 容易被字典攻击

**建议：**
添加盐值：
```cpp
static std::string getMD5WithSalt(QString password, QString salt)
{
    QString salted = password + salt;
    MD5 md(salted.toStdString());
    return md.toString();
}
```

#### D.3 缺乏输入验证

**问题描述：**
- 用户名、昵称等缺乏长度和格式验证
- 可能导致数据库注入（如果后端是SQL）

**建议：**
添加输入验证：
```cpp
bool validateTel(QString tel)
{
    // 验证手机号格式
    QRegExp rx("^1[3-9]\\d{9}$");
    return rx.exactMatch(tel);
}

bool validateName(QString name)
{
    // 验证昵称长度（1-20字符）
    return name.length() >= 1 && name.length() <= 20;
}
```

### ⚠️ F. 并发问题（严重性：中）

#### F.1 网络通信线程安全

**问题描述：**
- `TcpClient`使用单独的接收线程
- 但`DealData`直接更新UI，可能违反Qt的线程安全规则

**建议：**
使用信号-槽跨线程更新UI：
```cpp
void TcpClient::RecvThread(void* lpvoid)
{
    TcpClient* pThis = (TcpClient*)lpvoid;
    while(!pThis->m_isStop)
    {
        // 接收数据
        char* buf = new char[nlen];
        // ...
        // 使用信号-槽跨线程更新UI
        QMetaObject::invokeMethod(pThis->m_pMediator, "DealData",
                                  Qt::QueuedConnection,
                                  Q_ARG(unsigned int, lSendIP),
                                  Q_ARG(char*, buf),
                                  Q_ARG(int, nlen));
    }
}
```

## 六、改进建议总结

### 优先级分类

#### 🔴 高优先级（必须修复）
1. **修复内存泄漏**：完善`CKernel`的析构函数
2. **修复迭代器bug**：`rmmem()`函数中的编译错误

#### 🟡 中优先级（建议修复）
1. **消除硬编码**：使用常量或枚举
2. **改进错误处理**：添加返回值检查
3. **解耦设计**：减少模块间的依赖
4. **线程安全**：确保跨线程访问的安全性

#### 🟢 低优先级（可选优化）
1. **代码重构**：消除代码重复
2. **添加注释**：提高代码可读性
3. **单元测试**：添加测试用例
4. **使用智能指针**：提高内存安全性

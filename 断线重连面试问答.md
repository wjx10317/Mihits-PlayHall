## 一、房间与数据结构

### Q1: 单个房间最多加入多少人？

**2人参与对局，旁观不限人数。**

| 角色 | 枚举 | 人数 | 棋子 |
|------|------|------|------|
| 房主 | `_host` | 1 | 黑棋 |
| 玩家 | `_player` | 1 | 白棋 |
| 旁观 | `_spec` | 不限 | 只看不下 |

`userlist` 是 `std::list<QPair<int,QString>>`，无容量上限。UI 只有两个座位。

### Q2: 为什么用 `std::list` 而不是 `std::vector` 或 `unordered_map`？

| 操作 | `list` | `vector` | `unordered_map` |
|------|--------|----------|-----------------|
| 插入/删除 | O(1) | O(n) | O(1) |
| 遍历 | O(n) 缓存不友好 | O(n) 缓存友好 | O(n) |
| 按ID查找 | O(n) | O(n) | O(1) |

**实际选择理由**：频繁加入/退出房间，`list` 的 O(1) 插入删除合适。但说实话数据量小（几十人），用哪种差别不大。如果要优化，`unordered_map<int, QString>` 按 ID 查找更高效。

### Q3: 服务端用链表存房间用户，尾部旁观者最晚收到落子，有性能问题吗？

**链表串行广播确实存在尾部延迟偏高的问题。** 最坏情况下一个慢连接的 TCP 缓冲区满了会拖慢后面的发送。

**修复方案**：每次广播前拷贝到 `vector` 随机打乱顺序，或记录 `broadcast_offset` 从不同起点轮转。

**但实际影响可忽略**：五子棋落子频率极低（几十秒一着），单次消息体只有几个字节，非阻塞发送下微秒级差异无感知。

## 二、密码安全

### Q4: MD5 加密真的安全吗？

**不安全，三层硬伤：**

| 问题 | 后果 |
|------|------|
| MD5 已被碰撞破解（2004年） | Carnegie Mellon 宣布 MD5 "cryptographically broken" |
| 无盐 | 相同密码→相同密文，彩虹表秒破 |
| 单次哈希 | 现代 GPU 每秒数百亿次 MD5，6位数字密码 <1ms |

### Q5: 怎么改进？

| 方案 | 防彩虹表 | 防暴力 | 推荐度 |
|------|---------|--------|--------|
| MD5（现状） | ❌ | ❌ | — |
| MD5 + 固定盐 | △ | ❌ | — |
| SHA-256 + 随机盐 + PBKDF2 迭代 | ○ | △ | ○ |
| **bcrypt / scrypt / Argon2** | ✅ | ✅ | ✅ |

### Q6: 固定盐 vs 随机盐？

- **固定盐**：代码里写死 `SALT="mihits2024"`，攻击者拿到代码后重新跑彩虹表 → 自欺欺人
- **随机盐**：每个用户注册时独立生成随机字符串，不保密，明文存库。作用是把「算一次哈希暴打全库」变成「每个用户单独暴力破解」

### Q7: 盐的工作原理？

```
注册：
  salt = random_bytes(16)
  hash = SHA256(salt + password)
  存库: | salt(明文) | password_hash |

登录：
  查出该用户的 salt
  hash = SHA256(salt + 用户输入的password)
  比对 hash 和库里的 password_hash
```

**bcrypt 更简单**：盐和迭代次数编码在结果字符串里，调用方完全透明。

### Q8: TLS 怎么加？

项目是原生 Windows Socket API（`SOCKET`、`send()`、`recv()`），三条路线：

| 方案 | 改代码量 | 原理 |
|------|---------|------|
| **OpenSSL 包裹** | 中 | `SSL_new(CTX)` → `SSL_set_fd(sock)` → `SSL_write/SSL_read` 替换 `send/recv` |
| **Qt QSslSocket** | 大 | 把 `TcpClient/TcpServer` 从原生 Socket 换成 Qt 类 |
| **stunnel 代理** | 零 | 服务端前面放 stunnel，客户端连本地 stunnel，代码不动 |

## 三、对局记录

### Q9: 对局记录怎么存储？

MySQL `t_records` 表：

```sql
idA  idB  zoneid  roomid  data(JSON TEXT)  end_time(自动时间戳)
```

**完整流程**：
1. 开局 → `room_cache[roomid] = {}`
2. 每步落子 → `room_cache[roomid].push_back({x, y})`
3. 终局 → 序列化 `[{"x":7,"y":7},...]` → INSERT

### Q10: 为什么用 JSON？

| 考量 | JSON | 独立步骤表 |
|------|------|-----------|
| 存储 | 一个 TEXT 字段 | N 行记录，需 JOIN |
| 结构 | 灵活，无论步数 | 固定列 |
| 可读性 | 人眼可读 | 需工具解析 |
| 大小 | 225手约 3KB | 225行约 2KB（差距可忽略） |
| 解析 | Qt 自带 `QJsonDocument`，零依赖 | — |

**五子棋最多 225 步，不需要关系化。颜色不存，靠序号奇偶推断（0=黑, 1=白）。**

## 四、断线重连（Phase1 已实现）

### Q11: 项目实现了断线重连吗？

**没有。** 现状：
- `recv()==0` 时客户端能检测连接断开 → 抛 `SIG_disConnect` 信号 → **但没人接**
- `TcpClientMediator::SendData()` 有 socket 层重连能力 → **但没有会话恢复**
- 服务端只清 epoll fd → **没清 `m_mapfdtouserinfo` 和房间列表**
- 无心跳机制 → **无声断线（拔网线/WiFi掉）2小时 TCP keepalive 才感知**

### Q12: 双方都掉线了服务端怎么知道？

**不知道。** 当前没有心跳，服务端没有任何感知手段。`recv()` 阻塞等数据，TCP keepalive 默认 2 小时才触发。

### Q13: timer 在哪里？

**目前无回合 timer。** 若加，定为客户端自算（非服务端计时）。

### Q14: 掉线场景怎么处理？

| 场景 | 处理 |
|------|------|
| 一方掉线 + 有回合 timer | timer 到期 → 客户端本地判负 |
| 双方掉线 | 服务端无感知，等待 TCP keepalive(2h) |
| 掉线后重连 | 需回复 token → 服务端推送成员 + 棋局快照 |

### Q15: 重连需要单独定义协议吗？

**一套 `RECONNECT` 统一协议即可。** 不管在主页/分区/房间，本质相同：客户端发 token，服务端绑新 fd，返回当前状态（zoneid/roomid/inGame）。

### Q16: 旧 fd 释放了吗？

**已被网络层释放。** `recv_task` 里 `close(ev->fd)` 已关闭。但 `USER_INFO->m_sockfd` 还是旧值，重连时必须更新。

### Q17: epoll 怎么知道 socket 被关闭？

**内核自动处理。** `close(fd)` 时内核自动从所有注册的 epoll 实例摘除。不需要手动 `EPOLL_CTL_DEL`。

### Q18: `m_mapSockfdToEvent` 需要清理吗？

**心跳超时路径不能碰它**（耦合网络层内部），但为清零泄漏，通过 `Block_Epoll_Net::CloseFd(fd)` 公开 API 做 `eventdel + close`，ev 由 `recv_task` 自然清理路径回收（EPOLLONESHOT 保证无竞态）。

### Q19: token 存哪里？有并发问题吗？

存储：`USER_INFO::m_token[_MAX_SIZE]` + 客户端 `m_reconnectToken[_MAX_SIZE]`，登录时写一次，之后只读。

**`rand()` 有并发问题**：全局共享种子，多线程同时调会碰撞。已改为 `rand_r(&seed)`，每线程独立种子 = `pthread_self() ^ time(NULL)`。

## 六、Phase3 重连恢复（已实现）

### 服务端 `ReconnectRq`

1. 校验 userid + token
2. 绑定新 fd → userid，续心跳
3. 发 `RECONNECT_RS { result, zoneid, roomid, inGame }`
4. 若在房间：逐个发送 `STRU_ROOM_MENBER` → 终止标记 `{userid=-1}`
5. 若对局中：按 `room_cache` 顺序逐个发送 `STRU_FIL_PIECEDOWN`

### 客户端 `slot_reconnectRs`

1. `result=0` → 弹窗须重新登录
2. `zoneid==0` → 大厅
3. `zoneid>0, roomid==0` → 专区界面，拉专区信息
4. `roomid>0` → 清空房间 → `setinfo` → 复用 `slot_roommemberrq` 接收成员 → 复用 `slot_dealpiecedown` 接收落子

### 关键特性

- **终止标记**：`STRU_ROOM_MENBER{userid=-1}` 表示成员列表结束
- **零新协议**：成员和落子复用现有 `DEF_ROOM_MEMBER`、`DEF_FIL_PIECEDOWN`
- **两端同步**：双方都重新进房间 → 清除准备状态 → 重新准备 → 重新开始

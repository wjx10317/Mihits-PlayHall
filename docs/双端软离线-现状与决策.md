# 双端同时软离线 — 现状与决策

| 项 | 内容 |
|----|------|
| 状态 | **保持现有实现，暂不专项优化** |
| 日期 | 2026-07-21 |
| 关联 | 软断 / 硬断 / RECONNECT / `room_cache` |

---

## 1. 当前机制（摘要）

| 态 | 行为 |
|----|------|
| 软断 | `m_sockfd=-1`，保留 `USER_INFO` / token / `m_roomList` / `room_cache`；host/player 向**在线**成员发 `SOFT` |
| 重连 | token 校验 → 恢复层；对局重放落子；`REONLINE` 恢复头像 |
| 硬断 | 心跳超时（软离线 30s / 弱网在线 60s）→ 删会话；host/player 清 `room_cache` 并踢出 |
| 软断拒绝重连 | **不发 LOGOUT**，等硬超时清会话 |
| 主动退出（在线） | 发 `LOGOUT` → 硬断 |

SOFT 只发给 `SendToRoomOnline` 筛出的在线 fd；对方已软离线则**发不出去**。

---

## 2. 双端同时软离线会发生什么

### 2.1 共同现象

双方几乎同时断 TCP 时：

1. 各自 `OnFdClosed` → 软离线，会话与房间席位都保留  
2. 彼此的 `SOFT` 都发不出去（对方 fd 已无效）  
3. 两边 UI 都可能仍显示对方「在线彩色」，直到自己重连纠偏或对方硬超时  

### 2.2 分情况

| 场景 | 现有结果 | 严重度 |
|------|----------|--------|
| 都不在同一房间 | 互不影响，各自重连/超时 | 无 |
| 同房准备中，双双重连 | MEMBER / NOTREADY / REONLINE 可纠偏 | 轻（短暂假在线） |
| 同房准备中，一方重连一方挂机 | 重连方见「幽灵座位」约至硬超时 | 中 |
| 同房对局，双双重连 | 各自重放落子，可续局 | 轻 |
| 同房对局，一方重连一方硬超时 | 硬断清 `room_cache`，对局被拆 | 中～高（少见） |
| 双方都不重连 | ~30s 后依次 HARD，房间清空 | 可接受 |
| 仅 spec 软断 | 本就不广播 SOFT，对棋局影响小 | 无 |

「同时」比「先后几秒」更容易互相收不到置灰：后断一方若断前仍在线，可能收到先断方的 SOFT。

## 3. 已知可选方案（未做）

| 方案 | 作用 | 未做原因见下节 |
|------|------|----------------|
| 重连时按 peer `fd<0` 本地置灰 | 补「看不见对方掉线」 | 体验补丁，非刚需 |
| HARD 清 cache 条件收紧（仍有在线对局席则不清） | 减轻「一方回来一方超时拆局」 | 改变硬断语义，要再定产品规则 |
| 房间 `paused` / 双端离线态 | 结构化处理双断 | 协议与状态机变重 |
| 软断拒绝也发 LOGOUT | 缩短幽灵窗口 | 已否决：软断路径 LOGOUT 不可靠且与「等硬超时」策略冲突 |

---

## 4. 为什么保持现有

1. **主路径已闭环**  
   单端软断 → 重连 / 硬超时，以及正常离房、重登 `reset USER_INFO`，已按当前设计跑通。双端同时软离线是**低概率叠加场景**（同房 + 近乎同时断网 + 至少一方不及时重连）。

2. **SOFT「只通知在线」是刻意边界**  
   软断本意是「保留会话等本人重连」，不是可靠的双向在场同步。对方也软离线时发不出去，与「无 fd 不投递」一致；用重连 MEMBER / REONLINE / 硬超时收口，避免为离线信箱再加一层协议。

3. **硬超时清 `room_cache` 保护一致性**  
   对局席硬断即拆局、清缓存，保证不会长期保留无人认领的 `inGame` 热状态。双端一活一死后「活着的一方局被拆」是该语义的代价；若改为「有人在线就不清」，需重新定义「谁有权续局 / 判负 / 存谱」，成本高于当前收益。

4. **软断拒绝不发 LOGOUT 已定**  
   软断后 fd 映射已摘，LOGOUT 常无效；拒绝重连交给 30s 硬超时，逻辑简单、与心跳模型一致。

5. **优先稳定已修项，不为边角再扩状态机**  
   近期已收口：重登复用 `USER_INFO`、LeaveRoom 清 cache、P1 锁与 `continue`、软断不发 LOGOUT。双端软离线优化属于体验增强，**暂留**；P2（WIN 不广播）同样暂留。若线上同房双断投诉变多，再按「重连补置灰 → 再考虑 HARD 清 cache 条件」迭代。

---

## 5. 决策一句话

**承认双端同时软离线时可能短暂「互相看不见掉线」、以及「一侧重连一侧硬超时会拆局」；在主闭环已通、场景少见、改语义成本高的前提下，保持现有软/硬/重连模型，不做专项改造。**
